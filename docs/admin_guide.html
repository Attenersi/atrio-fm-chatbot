Instrukcja obsługi — Atrio FM Chatbot

Dla administratora — tylko to, co widać w przeglądarce. Bez wiedzy technicznej.

Canvas w Cursorze Interaktywny panel obok chatu zapisany jest w folderze projektu Cursora (nie w tym repozytorium Git). Zostaje na Twoim komputerze razem z projektem. Pełna treść masz też tutaj: zakładki poniżej oraz plik admin_guide.md (Markdown).

Atrio FM Chatbot to narzędzie do zarządzania budynkiem (Facility Management). System składa się z trzech głównych części dostępnych dla admina:

Chat z botem

Bot odpowiada na pytania o budynek i automatycznie tworzy zlecenia serwisowe (tickety), gdy wykryje usterkę.

Panel admina

Zarządzanie ticketami, użytkownikami, dokumentami wiedzy i lukami w wiedzy bota.

Panel jakości

Analiza błędów bota, ocena odpowiedzi i poprawa zachowania bez ingerencji w kod.

URL: /chat

Chat to rozmowa z botem FM. Bot klasyfikuje zgłoszenia i może automatycznie tworzyć tickety serwisowe.

Jak prowadzić rozmowę

  1. Otwórz /chat w lewym menu.
  2. Wpisz pytanie lub opis problemu w polu na dole.
  3. Wciśnij Enter lub kliknij Wyślij.
  4. Bot odpowiada na żywo — tekst pojawia się stopniowo.
  5. Pod odpowiedzią zobaczysz kategorię, priorytet i czy utworzono ticket.

Kiedy bot tworzy ticket

Tworzy ticket gdy

Problem wymaga technika; coś jest zepsute; dotyczy elektryki, hydrauliki, HVAC, wind; słowa jak awaria, przeciek, nie działa.

Nie tworzy ticketu gdy

Pytanie informacyjne (godziny, zasady); status istniejącego zgłoszenia; podziękowanie; ogólne procedury.

Priorytety

PriorytetKiedyPrzykład
URGENTBezpośrednie zagrożeniePożar, gaz, alarm, zalanie
HIGHPoważna usterka tego dniaBrak ogrzewania zimą, serwerownia
NORMALUciążliwe, nie krytyczneŻarówka, zacięte drzwi
LOWDrobna niedogodnośćWymiana mebla, estetyka

Przykłady

Zgłoszenie usterki

Wpisz:

„W toalecie damskiej na 3 piętrze ciągle kapie woda z kranu. Kapie od wczoraj.”

Oczekiwany efekt: Plumbing / NORMAL / ticket utworzony.

Pytanie informacyjne

Wpisz:

„Do której godziny jest otwarty budynek w weekendy?”

Oczekiwany efekt: odpowiedź z dokumentacji, bez ticketu.

Pilna awaria

Wpisz:

„Na korytarzu 2p widać dym i czuć spalony plastik przy gniazdku!”

Oczekiwany efekt: Electrical / URGENT / ticket + instrukcja bezpieczeństwa.

Create ticket anyway

Jeśli bot nie utworzył ticketu, a powinien — użyj przycisku pod odpowiedzią, aby zgłosić zlecenie ręcznie.

New Chat

Czyści historię i zaczyna nowy wątek. Używaj przy zmianie tematu — bot pamięta ostatnie ok. 60 wiadomości.

Limit Maks. ok. 30 wiadomości na minutę na użytkownika. Przy błędzie „Too Many Requests” odczekaj chwilę.
Czego nie robić
  • Nie ufaj samej odpowiedzi bota — sprawdź, czy ticket dotarł do technika.
  • Nie spamuj tym samym pytaniem — psuje kontekst rozmowy.
  • Przy URGENT nie polegaj tylko na chacie — zadzwoń na linię awaryjną.

URL: /dashboard

Lista ticketów. Admin widzi wszystkie; zwykły użytkownik tylko swoje.

Filtrowanie

  1. Otwórz /dashboard.
  2. Użyj filtrów: kategoria, priorytet, status (np. URGENT + Open).

Zmiana statusu

  1. Kliknij ticket w tabeli.
  2. W szczegółach wybierz status i zapisz.
StatusZnaczenieKiedy ustawić
OpenNowe, nikt nie zacząłDomyślnie
In ProgressTechnik pracujePo przekazaniu zadania
ResolvedRozwiązanePo potwierdzeniu naprawy

Notatki serwisowe

W /admin → Tickets: Add Resolution Note — opis pracy, części, koszt, czas (minuty).

Korekta klasyfikacji

  1. W /admin znajdź ticket → Classification Override.
  2. Wybierz pole: category, priority lub department.
  3. Wpisz poprawną wartość i zatwierdź — zapisuje się historia i dane treningowe.

Eksport CSV: eksportuje to, co widzisz po filtrach.

Czego nie robić
  • Nie ustawiaj Resolved bez potwierdzenia od technika.
  • Nie ignoruj URGENT — reaguj w ciągu godzin.

URL: /admin → Knowledge Gaps

Lista pytań, na które bot nie znalazł odpowiedzi w dokumentacji. Każda luka = sygnał do uzupełnienia wiedzy.

Dlaczego to ważne Uzupełnianie luk poprawia odpowiedzi dla wszystkich użytkowników — nie tylko dla jednej rozmowy.

Jak uzupełnić lukę

  1. Otwórz lukę lub stronę szczegółów luki.
  2. Doc name — np. building_hours.md.
  3. Content — treść po angielsku (bot pracuje po angielsku).
  4. append dodaje do pliku; overwrite zastępuje cały plik.
  5. Zaznacz Auto-reindex, żeby bot od razu widział zmianę.
  6. Resolve Gap — luka zamknięta, plik zapisany.

Przykłady

Godziny otwarcia

Pytanie: „What time does the building close on Saturdays?”

Doc: building_hours.md, tryb append

## Building Opening Hours

Monday–Friday: 7:00 AM – 10:00 PM
Saturday: 8:00 AM – 6:00 PM
Sunday: Closed

Security desk is always staffed 24/7.
Parkowanie gości

Pytanie: „Where can visitors park?”

Doc: 07_parking_transport.md, tryb append

## Visitor Parking

Visitor parking is available in Level B1. Visitors must register at reception.
Maximum 4 hours. Barrier code available from the reception desk.
Czego nie robić
  • Nie pisz treści po polsku w dokumentach dla bota.
  • Nie używaj overwrite, jeśli nie wiesz, co jest w pliku.
  • Nie pomijaj Auto-reindex — bez tego bot nie widzi nowej treści.

URL: /admin/training-quality

Analiza błędów bota, testy jakości i zatwierdzanie zmian w instrukcjach (bez kodu).

Jak to działa System grupuje błędy → AI proponuje poprawkę tekstu instrukcji → Ty akceptujesz lub edytujesz → bot od razu korzysta z nowej reguły.

1. Mismatch Groups

Grupy błędów (kategoria, priorytet, ticket, brak informacji w odpowiedzi) — widok bez dodatkowego AI.

2. Eval Runs

  1. Run Eval — automatyczny test na zestawie ~80 pytań (kilka minut).
  2. Status: running → done; wynik: accuracy ogólna i per pole.
  3. Tylko jeden eval naraz — nie klikaj drugi raz, dopóki pierwszy się nie skończy.

3. Suggested Fixes

  1. Analyze Pending — propozycje tekstu do instrukcji (limit: ok. 1 na 5 minut; wynik cache'owany ~24 h).
  2. Przeczytaj sugestię → Apply lub edytuj w modalu.

4. Active Overrides

Lista aktywnych reguł, delta accuracy (przed/po), przycisk Rollback gdy coś pogorszyło wyniki.

Przykładowy workflow

  1. Widzisz dużo błędów kategorii (np. kondensacja → zły dział).
  2. Uruchom Run Eval i zapisz wynik (baseline).
  3. Analyze Pending → przeczytaj sugestię.
  4. Apply jeśli ma sens.
  5. Później ponownie Run Eval — sprawdź, czy accuracy wzrosła.
  6. Jeśli spadła — Rollback.
OgraniczenieCo robić
Max 5 aktywnych overridesRollback starych przed nową
Świeży eval przed ApplyUruchom Run Eval, jeśli system tego wymaga
Niska pewność sugestiiEdytuj tekst ręcznie przed zatwierdzeniem
Czego nie robić
  • Nie akceptuj sugestii, których nie rozumiesz.
  • Nie ignoruj spadku accuracy — użyj Rollback.
  • Unikaj ciągłego Eval w godzinach szczytu — obciąża API i może spowolnić chat.
